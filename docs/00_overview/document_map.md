# Document Map

このプロジェクトでは、**ドキュメント自体にもアーキテクチャを持たせる**。  
仕様、データ設計、アーキテクチャ、実装指針、将来構想を混在させない。

---

## ディレクトリ構成

### 00_overview
プロジェクト全体像を示す。  
初めて読む人が最初に読む場所。

### 01_product_spec
ユーザー視点で「アプリが何をするか」を定義する。  
UI やコード構造ではなく、挙動と仕様を記述する。

### 02_domain
データ設計を扱う。  
型、用語、構造、状態モデルを記述する。

### 03_architecture
レイヤ分離、依存方向、責務分担を扱う。

### 04_usecases
操作ごとの状態遷移やユースケース責務を扱う。

### 05_ui
画面構成、各画面の役割、表示モデルの意味を扱う。

### 06_persistence
保存方式、永続化方針、プリセット管理を扱う。

### 07_engine
タイマーエンジンのようなコア挙動を扱う。

### 08_guides
開発者向けガイド。どう変更すべきか、どう実装すべきかを書く。

### 09_roadmap
未確定の将来構想。実装対象ではない。

### 10_adr
重要な設計判断の履歴。  
「なぜこうしたか」を残す。

---

## どのドキュメントを更新すべきか

### ユーザー仕様が変わった
- `01_product_spec/*`

### 型やモデルが変わった
- `02_domain/*`

### レイヤ責務が変わった
- `03_architecture/*`

### 振る舞いが変わった
- `04_usecases/*`
- 必要に応じて `07_engine/*`

### UI の役割や画面が変わった
- `05_ui/*`

### 保存方法が変わった
- `06_persistence/*`

### 将来構想を追加したい
- `09_roadmap/*`

### 設計判断を記録したい
- `10_adr/*`

---

## ドキュメントを書くときの原則

1. 仕様と実装詳細を混ぜない
2. 確定仕様と将来構想を混ぜない
3. 現在のコードに存在する責務分離と整合させる
4. 「なぜ」を残したい場合は ADR に書く
5. コードを読まなくても全体像が分かる密度を保つ
