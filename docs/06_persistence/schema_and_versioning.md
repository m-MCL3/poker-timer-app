# Schema and Versioning

## 1. 現方針

モック段階では複雑な migration を持たない。  
**現時点の構造を v1 とし、それ以前の互換は切り捨てる。**

---

## 2. この方針を採る理由

- 仕様がまだ動いている段階で migration を増やすとノイズになる
- 旧モデルの存在が新設計の理解を阻害する
- まず現行構造を固める方が価値が高い

---

## 3. 将来の拡張余地

将来的に保存形式を変更する場合は、

- schemaVersion を上げる
- deserialize 側で分岐する
- 必要なら migrate を導入する

ただし、そのタイミングはモック脱却後とする。
