# 変更方針

## 1. 基本姿勢

このプロジェクトでは、場当たり的な修正よりも、理想アーキテクチャに寄せる変更を優先する。  
ただし、一度に全てを変えるのではなく、責務境界を崩さない単位で進める。

## 2. 優先順位

1. Domain の語彙と正規形を守る
2. Usecase に振る舞いを集約する
3. UI を薄く保つ
4. Infrastructure の都合を内側へ漏らさない

## 3. 変更時の確認項目

- docs は更新したか
- naming は既存ルールに揃っているか
- Structure / Runtime / Snapshot の境界を壊していないか
- UI が過剰にロジックを持っていないか
- ADR を追加すべき判断か
